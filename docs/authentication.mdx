By leveraging OAuth 2.0, developers can empower their applications to interact with the Alvys API on behalf of their users. This guide outlines the authentication process and provides instructions for obtaining access tokens through direct application integration.

<Note>
  **Getting API access**

  - Existing Alvys customers can obtain API access by contacting their account representative.
  - Independent Software Vendors (ISVs) should contact the Alvys Partnership team.
</Note>

## Creating client application credentials

Follow these steps to create your credentials in the Alvys Admin Portal:

1. Navigate to **Admin → API Access**
2. Click **Create New Application**
3. Fill out the **Name** and **Description**
4. Select your desired **permissions** (scopes)
5. Set an optional **expiration date** for the credentials
6. Click **Generate**

After creation, your **Client ID** and **Client Secret** are shown along with the scopes included for token generation. Customers can generate up to 10 sets of client credentials; only newly generated ones can be edited.

<Warning>
  Client ID and Client Secret are sensitive. Store them securely — never share them publicly or expose them in front-end code.
</Warning>

## Token endpoint

All new token requests must use:

```
https://auth.alvys.com/oauth/token
```

### Request body (JSON)

```json
{
  "client_id": "YOUR_CLIENT_ID",
  "client_secret": "YOUR_CLIENT_SECRET",
  "audience": "https://api.alvys.com/public/",
  "grant_type": "client_credentials"
}
```

<Note>
  When including a `scope` field in the token request:

  - The returned token **always includes all scopes** granted to your client application, regardless of what you specify.
  - Including a `scope` field does **not** override or limit access — it only causes the request to fail if you include a scope not granted to your client.
  - If `scope` is omitted, the token still includes **all granted scopes**.
</Note>

### cURL example (JSON)

```bash
curl --request POST \
     --url 'https://auth.alvys.com/oauth/token' \
     --header 'Content-Type: application/json' \
     --data '{
       "client_id": "YOUR_CLIENT_ID",
       "client_secret": "YOUR_CLIENT_SECRET",
       "audience": "https://api.alvys.com/public/",
       "grant_type": "client_credentials"
     }'
```

### cURL example (form-encoded)

```bash
curl --request POST \
     --url https://auth.alvys.com/oauth/token \
     --header 'Content-Type: application/x-www-form-urlencoded' \
     --data 'client_id=YOUR_CLIENT_ID' \
     --data 'client_secret=YOUR_CLIENT_SECRET' \
     --data 'audience=https://api.alvys.com/public/' \
     --data 'grant_type=client_credentials'
```

Use the resulting access token in API request headers:

```
Authorization: Bearer YOUR_ACCESS_TOKEN
```

Each client credential's token is restricted to the exact scopes you assign in **Admin → API Access**.

<Warning>
  While you can assign `create`, `update`, and `delete` scopes and your token will include them, the corresponding **write endpoints are not yet available** for all resources. The Public API currently exposes **read** operations broadly; additional write scopes are in place for seamless adoption as write functionality is released.
</Warning>

## Scope claim

Legacy tokens from the previous `/api/authentication/{tenant_id}/token` flow do not include the `scope` claim. Migrate to the new flow to avoid disruption.

New tokens follow a fine-grained permissions model:

```
"scope": "load:read driver:create"
```

Scopes control access to API endpoints and must be selected when creating your application.

## Troubleshooting

- Double-check your `client_id`, `client_secret`, and `audience`
- Ensure scopes are correctly assigned in **Admin → API Access**
- Validate that the client is active and not expired
- Use only `application/json` or `application/x-www-form-urlencoded` content types
- Technical support: [support@alvys.com](mailto:support@alvys.com)
